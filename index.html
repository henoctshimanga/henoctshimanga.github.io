SmartCourse Finder

From a C++ console application to a full-stack system with modern architecture, database integration, and graph-based logic.

SmartCourse Finder is a course management and advising system designed to help users explore course offerings, manage prerequisites, and generate valid course completion paths. The system allows users to import course data, search courses, view prerequisite relationships, and interact with structured academic planning features. This portfolio demonstrates how a simple data-driven application can evolve into a scalable system by integrating frontend, backend, and database layers.

Artifact Overview

Main Project

Original Version

Built as a C++ console-based application, the original SmartCourse Finder loaded course data from files, stored information in memory, and provided basic search and display functionality.

  • Built in C++ as a console-based application
  • Loaded course data from a file
  • Stored data in memory
  • Basic search and display functionality
  • Limited scalability and no persistence

Enhanced Version

Transformed into a full-stack application using React and Spring Boot, featuring a modern REST API, persistent MySQL database, and graph-based prerequisite management with improved UI/UX.

  • Full-stack application using React and Spring Boot
  • REST API for backend communication
  • MySQL database with JPA/Hibernate
  • Graph-based prerequisite management
  • Supports CRUD operations and course validation
  • Improved UI/UX with theme toggle and state persistence

Project Evolution

  1. Initial prototype: C++ console-based course advising system.
  2. Full-stack redesign: Separated frontend and backend for maintainability.
  3. Algorithm enhancement: Added graph-based prerequisite traversal and topological ordering.
  4. Database migration: Moved from H2 to MySQL with JPA/Hibernate persistence.
Software Design & Architecture

Design, Engineering & Architecture

Design Principles

The original C++ console design was a functional foundation, but the enhanced solution required a more sustainable architecture. The full-stack redesign introduced stronger separation of concerns, better usability, and a maintainable codebase.

  • Original C++ design used file flow and text output for simple course advising.
  • Transitioned to a full-stack web application to support real deployment and UX improvements.
  • Separated frontend and backend logic to improve maintainability and extensibility.
  • Introduced reusable, modular components and API boundaries for cleaner engineering.
  • Added usability improvements such as a modern interface and theme toggle for preference support.
Original Code Enhanced Code Narrative Document

Layered Architecture

The enhanced system follows a layered full-stack architecture that separates concerns across presentation, business logic, and data layers.

  • Frontend (React): Handles user interaction, state management, and UI rendering
  • Backend (Spring Boot): Provides RESTful APIs, business logic, and validation
  • Database (MySQL): Stores course and prerequisite data persistently
  • Application Logic: Uses graph-based structures to manage relationships and enforce rules

This separation of concerns improves scalability, maintainability, and testability across the entire system.

Algorithms

Data Structures & Logic

The original flat lookup approach was sufficient for basic search but not for modeling course dependency complexity. The enhanced artifact uses graph-based structures to better represent prerequisites and support advanced validation.

  • Flat lookup could not easily support multi-step prerequisite resolution or cycle detection.
  • Graph-based course relationships capture dependencies naturally and support traversal.
  • Prerequisite traversal validates course sequences and supports logical progression.
  • Topological ordering helps identify valid course pathways and detect dependency cycles.
  • The enhanced logic is more structured, maintainable, and aligned with advising needs.

Databases

Persistence & Schema

The persistence layer matured from H2 testing to MySQL with Hibernate/JPA. This progression highlights the difference between structured data storage and the application layer, which enforces graph-based prerequisite rules.

  • The initial H2 implementation supported rapid iteration and backend validation.
  • Migration to MySQL introduced durable storage, schema management, and production relevance.
  • JPA/Hibernate persistence maps course entities and prerequisite relationships with auto-generated schema.
  • The current schema includes courses and course_prerequisites tables with validation for required fields and order.
  • The database stores structured records while graph logic enforces advanced prerequisite validation.
  • Resolving Hibernate/MySQL configuration issues reinforced proper persistence setup and compatibility awareness.

Algorithms & Data Structures

The original implementation used a flat lookup structure, which was adequate for simple course search but limited the system’s ability to manage complex prerequisite relationships. The enhanced artifact adopts graph-based handling to model courses and prerequisites more naturally.

  • Flat lookup designs were easy to implement but struggled with dependency traversal and validation.
  • Graph-based course relationships support richer prerequisite logic and dependency analysis.
  • Prerequisite traversal enables validation of course sequences and supports navigation of course dependencies.
  • Topological ordering helps determine valid course progression and identify cycles.
  • The enhanced logic is more structured, maintainable, and aligned with real advising requirements.
Artifact Code Narrative Document

Databases

The database journey began with H2 in-memory storage for initial testing and moved to MySQL with JPA/Hibernate for production-grade persistence. The design separates database storage from application-level graph logic so the database holds structured records while the application enforces advanced prerequisite rules.

  • Initial H2 implementation allowed rapid prototyping and backend testing.
  • Migration to MySQL introduced durable persistence, real schema management, and production relevance.
  • Hibernate/JPA persistence supports course and prerequisite mappings with auto-generated schema.
  • The schema includes `courses` and `course_prerequisites` tables with validation for course existence and ordering.
  • Database storage is distinct from graph-based application logic, which governs advanced prerequisite validation and cycle detection.
  • I also resolved a Hibernate/MySQL configuration challenge, reinforcing the importance of correct dialect and persistence settings.
